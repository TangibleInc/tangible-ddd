# Consumer Modules and Sidecar Design

**Status:** Approved 0.6.2 production contract
**Target release:** Tangible DDD 0.6.2
**Depends on:** Tangible DDD 0.6.1 compiled `LongProcessCatalog`
**Date:** 2026-07-21

## Problem

A separately deployed WordPress plugin must be able to add DDD behavior to an
existing consumer without becoming a second DDD consumer. For example, a
sidecar may define classes below
`Tangible\LMS\Extension\SuperTrace\...`, while LMS remains the owner of the
DDD identity:

- commands and queries resolve handlers and dependencies from the sidecar's
  container;
- domain and integration events use the LMS action prefix;
- command audit, correlation, outbox, process, option, and Action Scheduler
  writes use the LMS configuration and tables;
- `LongProcess` rows and continuations stay in the LMS process lane; and
- the dashboard lists LMS once and shows sidecar class names inside LMS data.

LMS and Quiz release builds may load immutable Symfony containers generated by
`PhpDumper`. A sidecar cannot append definitions, tags, handlers, or compiler
passes to those containers after release. The extension mechanism therefore
must not depend on a retained `ContainerBuilder` or on editing a generated
container.

Notifications is not a DDD consumer. Nothing in this design infers ownership
from a broad Tangible namespace, an active plugin, or a table-name guess. A
module is attributed only through an explicit registration against an already
registered DDD consumer and a namespace strictly below that consumer's root.

## Current Runtime Facts

The existing framework already separates two axes that a module needs:

1. `ConsumerRegistry` stores top-level consumer discovery handles by DDD
   prefix.
2. `ConsumerRegistry::owner_of()` routes a concrete class by longest matching
   namespace root.
3. `CommandBusAware` and `QueryBusAware` obtain the container from that owner.
4. `Event::prefix()` obtains the prefix from that owner.
5. A `ConsumerHandle` already carries config and container independently.
6. `consumers()` and the dashboard read `ConsumerRegistry::all()`, not
   `owner_of()`'s internal routing candidates.

The missing capability is a routing candidate whose config comes from a host
consumer while its container comes from a module, without inserting that
candidate into the top-level consumer map.

Process discovery is a separate lifecycle concern. Tangible DDD 0.6.1 will
compile host `ddd.long_process` tags into an immutable `LongProcessCatalog` and
will register that catalog before falling back to retained-builder discovery.
Tangible DDD 0.6.2 must layer module process entries over that base; it must not
replace or redesign the compiler pass.

## Goals

- Route module classes through a module-owned DI container.
- Reuse the exact registered host `IDDDConfig` and therefore its prefix and
  persistence identity.
- Keep top-level consumer and dashboard discovery unchanged.
- Work when the host container is an opaque dumped Symfony container.
- Give module handlers and integration listeners a normal eager-boot path.
- Register module process types on the host `ProcessRunner` after the 0.6.1
  compiled catalog.
- Fail loudly on missing hosts, cross-consumer namespaces, ambiguous roots,
  late registration, or broken module wiring.
- Keep the core mechanism generic; Super Trace is an example namespace, not a
  framework feature.

## Non-Goals

- Mutating a host container or generated PHP container.
- Giving a module its own DDD tables, migrations, outbox worker, continuation
  hooks, or dashboard consumer entry.
- Automatically treating arbitrary WordPress plugins as DDD consumers.
- Scanning broad namespaces or plugin directories to infer modules.
- Making Notifications a DDD consumer or silently attributing its classes to
  LMS.
- Implementing the Super Trace business scenario.
- Changing the 0.6.1 compiler pass, catalog, or catalog-first hook logic.
- Supporting safe sidecar deactivation while its process rows are live without
  an explicit drain or migration procedure.

## Architecture Options

### Option 1: A child consumer in the top-level registry

Create a second `ConsumerHandle` with the host config and module container, then
store it as a normal consumer.

Advantages:

- Reuses the existing handle and routing code.
- Requires little new runtime machinery.

Rejected because:

- top-level consumers are keyed by prefix, so using the host prefix replaces
  the host handle;
- inventing a second prefix creates false tables, hooks, migrations, and
  dashboard identity;
- exposing the child through `all()` makes discovery surfaces treat it as an
  independent persistence universe; and
- a parent/child object graph does not itself solve late process registration.

### Option 2: Mutate the host container

Have the sidecar add service definitions and tags to the host
`ContainerBuilder` before or after host compilation.

Advantages:

- One container and one bus after a successful pre-compile mutation.
- Existing handler and tag discovery could remain unchanged.

Rejected because:

- a dumped production container is not a `ContainerBuilder` and exposes no
  definition or tag API;
- plugin load order becomes a hidden build requirement;
- release behavior differs from debug behavior;
- post-compile mutation is unsupported by Symfony; and
- hand-editing or regenerating a host release artifact defeats separate
  deployment.

### Option 3: A separate runtime module-routing overlay

Keep top-level consumers untouched. Store module routing handles in a second
map keyed by module namespace root. Each handle receives the registered host
config and the module container getter. `owner_of()` considers top-level and
module roots together and keeps longest-root-wins resolution. `all()` continues
to return top-level consumers only.

Advantages:

- directly matches the existing split between namespace routing and prefix
  discovery;
- does not mutate either container;
- works with retained, compiled, and dumped host containers;
- keeps dashboard and migration identity singular; and
- creates a natural enumeration seam for post-0.6.1 process overlays.

Cost:

- the module must build or ship its own container and wire its bus correctly;
- host-shared infrastructure services must be referenced deliberately; and
- module hook/process boot happens in a separate, ordered runtime phase.

**Decision:** Option 3.

## Core Invariants

1. A module has one explicit host prefix.
2. That host must already exist in the top-level consumer registry.
3. The module root must be a strict, whole-segment descendant of the host root.
4. The named host must be the most specific top-level consumer that contains
   the module root. A module cannot bypass a nested consumer by naming a broader
   parent.
5. A module handle uses the host's exact `IDDDConfig` object.
6. A module handle resolves `container()` through the module getter.
7. `owner_of()` uses the longest matching root across consumers and modules.
8. `ConsumerRegistry::all()` never includes module handles.
9. Modules never register migrations, outbox workers, process continuation
   hooks, or a second dashboard consumer.
10. Module process entries are installed on the host `ProcessRunner`, never a
    parallel runner.
11. No API mutates the host container.
12. Once a module is attached, later top-level consumer registrations cannot
    tie or outrank its original host or partition the module namespace.

## Registry API

The low-level framework API is:

```php
namespace TangibleDDD\Infra\Consumers;

final class ConsumerRegistry
{
    public static function consumer(string $prefix): ConsumerHandle;

    public static function config_for(string $prefix): IDDDConfig;

    public static function service_for(string $prefix, string $service_id): mixed;

    public static function add_module(
        string $host_prefix,
        string $namespace_root,
        callable $di_getter,
    ): ConsumerHandle;

    /** @return array<string, ConsumerHandle> namespace root => routing handle */
    public static function modules_for(string $host_prefix): array;
}
```

`add_module()` creates an ordinary `ConsumerHandle` from:

- `config`: `consumer($host_prefix)->config()`;
- `di_getter`: the sidecar's getter;
- `label`: the host label; and
- `namespace_root`: the explicit module root.

The handle is stored only in a private module overlay with its host prefix.
This preserves the existing `owner_of(): ConsumerHandle` return contract.

After the first module attaches, the top-level host handle is stable. An exact
repeat of the host registration is idempotent; a different config, getter,
label, or namespace root throws. This prevents module handles from retaining
one host config while `service_for()` begins resolving a replacement host
container. A later top-level consumer also fails if its namespace root would
tie or outrank the module's original host at the module root, or if it would
claim a subtree below the module root. Broader ancestors and disjoint roots do
not change module ownership and remain valid.

`add_module()` is a low-level pre-runtime API. Its same-host replacement is
needed by the facade's documented pre-`init` getter replacement. The registry
does not know WordPress lifecycle state, so callers must use `boot_module()` in
WordPress. A direct post-wiring `add_module()` call is unsupported; preventing
that call would require a new public bypass/token or cross-layer lifecycle
coupling and is explicitly deferred from 0.6.2.

`config_for()` is both a convenience and a Symfony factory seam. A runtime
module container can receive the object directly. A separately dumped module
container can define its `IDDDConfig` service with a runtime factory that calls
`ConsumerRegistry::config_for('tgbl_lms')`; the dumped definition remains
immutable while the service resolves the registered host object when used.

`service_for()` is the cross-container service bridge. It resolves the host
handle, obtains the host runtime container, and calls only its normal `get()`
API. It never exposes a `ContainerBuilder`, definitions, or mutation. A dumped
module container uses this static method as a factory for the host's actual
stateful middleware and repositories. Returning the actual object, rather than
constructing an equivalent service from `config_for()`, preserves the host
transaction manager, `EventsUnitOfWork`, outbox implementation, and runner
state.

The architectural spike implements these registry methods and routing tests.
The WordPress facade below waits for the 0.6.1 rebase because its process path
must call the shared catalog-entry registration function introduced there.

## WordPress Facade

After rebasing onto 0.6.1, 0.6.2 adds a facade equivalent to:

```php
add_action('plugins_loaded', static function (): void {
    \TangibleDDD\WordPress\boot_module(
        host_prefix: 'tgbl_lms',
        namespace_root: 'Tangible\LMS\Extension\SuperTrace',
        di_getter: static fn () => \Tangible\LMS\Extension\SuperTrace\DI\di(),
    );
}, 30);
```

`boot_module()` is intentionally not a second `boot()`:

- it does not call `ConsumerRegistry::add()`;
- it does not register migrations;
- it does not register outbox or process-continuation workers; and
- it does not install a top-level discovery handle.

The sidecar must load its own compatible `tangible/ddd:^0.6.2` Composer copy
from its main plugin file before registering this callback. That copy must
participate in the loader's `plugins_loaded:0` version registration. The
newest winner initializes at priority 1, host consumers boot after that and
before priority 30 (priority 10 is the current fleet convention), and the
sidecar calls `boot_module()` at priority 30. The sidecar must not
autoload `TangibleDDD\...` runtime classes before winner initialization;
otherwise an older first-autoloaded class can defeat the newest-winner class
loader even though version registration succeeded.

`boot_module()` registers the module overlay after the host has booted and
wires module runtime services after host hooks have registered. Calls install
one `init:3` callback per normalized module root. A second call for the same
root and host replaces the getter only before WordPress `init` begins; a
conflicting host fails. Any call after WordPress `init` has begun fails before
installing a registry route or callback, even if priority 3 has not run yet.
Successful registration calls
`Tangible_DDD_Versions::instance()->require_version()` with minimum `0.6.2`
under the stable identifier `ddd-module:<normalized namespace root>`, so an
unmet winner is visible to diagnostics without inventing a second consumer
prefix.

## Lifecycle and Ordering

The supported WordPress order is exact:

1. The host and sidecar main files each load their bundled Composer autoloader.
   Both compatible Tangible DDD copies register with the newest-wins loader at
   `plugins_loaded` priority 0. The sidecar requires at least 0.6.2.
2. The winning Tangible DDD copy initializes at `plugins_loaded` priority 1 and
   installs its class/procedural runtime.
3. The host plugin calls `boot()` after winner initialization and before
   priority 30. Priority 10 is the current LMS, Quiz, and Cred convention;
   generated wiring must follow the convention finalized by 0.6.1. `boot()`
   immediately inserts the host in `ConsumerRegistry`.
4. The sidecar calls `boot_module()` at `plugins_loaded` priority 30.
   `boot_module()` resolves the named host and calls
   `ConsumerRegistry::add_module()`. This makes namespace routing available
   before container compilation and fails before normal requests if the host
   is absent.
5. The host and sidecar independently construct or load their containers. Any
   runtime compilation occurs on `init` priority 1. Neither container is
   mutated by the other.
6. Host `register_hooks()` runs on `init` priority 2. On 0.6.1 it registers the
   host compiled `LongProcessCatalog`, or the retained-builder fallback.
7. Module runtime wiring runs on `init` priority 3:
   - validate that module `IDDDConfig` is the host's exact object and that the
     module exposes a real `League\Tactician\CommandBus`;
   - read and validate the module container's `LongProcessCatalog`, when
     present;
   - eagerly instantiate module event handlers and integration listeners from
     the module container, propagating construction failures; and
   - register those entries on the host container's `ProcessRunner` through
     the shared 0.6.1 entry-registration function.
8. REST, admin, CLI, Action Scheduler, and domain work run after module routing
   and hooks exist.

A host that only calls `register_hooks()` at `init:2` is too late for step 4.
Module-capable hosts must use `boot()` so their identity exists at
`plugins_loaded:30`. This is an explicit compatibility requirement, not a
silent retry.

## Module Container Contract

The module container owns module application services, handlers, listeners,
and its command/query buses. It must expose at least:

- `League\Tactician\CommandBus`;
- `tactician.query_bus` if module queries use `send()`;
- public handler/listener services needed by eager boot; and
- `LongProcessCatalog` if the module declares process types.

The module's `IDDDConfig` service must be the host object returned by
`ConsumerRegistry::config_for($host_prefix)`. Identity alone is not enough for
host-native execution: a second `CorrelationMiddleware`, transaction manager,
`EventsUnitOfWork`, outbox repository, or `ProcessRunner` would fork state even
with the same prefix. The module therefore imports the host's actual public
instances through `ConsumerRegistry::service_for()`:

- `CorrelationMiddleware`;
- the host transaction middleware, using its host-specific service ID;
- `DomainEventsPublishMiddleware` and its `EventsUnitOfWork`;
- `IIntegrationEventBus` and `IOutboxRepository`; and
- `ProcessRunner`.

At `init:3`, the facade resolves and validates the mandatory `IDDDConfig` and
`League\Tactician\CommandBus` services before constructing any listener. A
missing service, a different config object, or a non-`CommandBus` value fails
module boot. Listener construction failures propagate and leave the module
unwired; the top-level consumer helper retains its historical log-and-continue
behavior.

The current LMS, Quiz, Cred, and Datastream service YAMLs set
`_defaults.public: true`. The six non-transaction IDs above are common across
those hosts. Transaction is not: Cred and Datastream command buses use
`TangibleDDD\Application\Persistence\TransactionMiddleware`, while LMS and
Quiz command buses use their own `DoctrineTransactionMiddleware` services.
The exact transaction ID is therefore part of the host/module contract, not a
framework-wide default.

Only the terminal command/query resolution layer is module-container bound so
it can resolve module handlers and self-handling command dependencies.

An independently dumped module container can use exact Symfony YAML like:

```yaml
services:
  _defaults:
    autowire: true
    autoconfigure: true
    public: true

  TangibleDDD\Infra\IDDDConfig:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'config_for']
    arguments: ['tgbl_lms']

  TangibleDDD\Application\Correlation\CorrelationMiddleware:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'TangibleDDD\Application\Correlation\CorrelationMiddleware'

  super_trace.host_transaction_middleware:
    class: League\Tactician\Middleware
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'Tangible\LMS\Application\Middleware\DoctrineTransactionMiddleware'

  TangibleDDD\Application\Events\DomainEventsPublishMiddleware:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'TangibleDDD\Application\Events\DomainEventsPublishMiddleware'

  TangibleDDD\Application\Events\EventsUnitOfWork:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'TangibleDDD\Application\Events\EventsUnitOfWork'

  TangibleDDD\Application\Events\IIntegrationEventBus:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'TangibleDDD\Application\Events\IIntegrationEventBus'

  TangibleDDD\Infra\IOutboxRepository:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments: ['tgbl_lms', 'TangibleDDD\Infra\IOutboxRepository']

  TangibleDDD\Application\Process\ProcessRunner:
    factory: ['TangibleDDD\Infra\Consumers\ConsumerRegistry', 'service_for']
    arguments:
      - 'tgbl_lms'
      - 'TangibleDDD\Application\Process\ProcessRunner'

  League\Tactician\CommandBus:
    arguments:
      - '@TangibleDDD\Application\Correlation\CorrelationMiddleware'
      - '@super_trace.host_transaction_middleware'
      - '@TangibleDDD\Application\Events\DomainEventsPublishMiddleware'
      - '@TangibleDDD\Application\CQRS\SelfExecutingCommandMiddleware'
      - '@tactician.middleware.command_handler'

  TangibleDDD\Application\CQRS\SelfExecutingCommandMiddleware:
    arguments: ['@service_container']
```

The host-specific transaction service ID is an explicit sidecar dependency.
It must be declared in the sidecar wiring and covered by a dumped-container
contract test. The bridge calls only `get()` on that opaque container; it never
mutates it.

If a host does not expose a required custom infrastructure service, full
host-native semantics are blocked until the host publishes a stable public
service ID in a normal release. Constructing an equivalent service from
`config_for()` is not an accepted substitute for stateful middleware,
transaction, unit-of-work, outbox, or process services.

## Command, Query, and Event Flow

For a module command class:

1. `CommandBusAware::container()` calls `owner_of(static::class)`.
2. The module root wins over the shorter host root.
3. The module handle returns the module container.
4. The module bus resolves the module handler or self-handling dependencies.
5. Host-scoped correlation/audit middleware writes to the host command-audit
   table.

Queries follow the same ownership route but use the module query bus.

For a module event class:

1. `Event::prefix()` calls `owner_of(static::class)->prefix()`.
2. The module handle delegates identity to the host config.
3. Domain and integration action names therefore use the host prefix.
4. A module integration listener is instantiated by the module container.
5. Its translated command routes back through the module handle and module bus.

The framework does not rewrite class names. Audit, outbox, and process rows keep
the module FQCN while living in host tables, which is the desired attribution.

## LongProcess Overlay on 0.6.1

The 0.6.1 host catalog is the immutable base. Module catalogs are runtime
additions:

```text
host compiled LongProcessCatalog
    -> register_process_entries(host config, host runner, base entries)
    -> for each registered module of this host, in registration order:
         module LongProcessCatalog
           -> validate module-owned class roots
           -> register_process_entries(host config, host runner, module entries)
```

Important properties:

- the host catalog object is never changed;
- the host container is never changed;
- all registrations target the one host `ProcessRunner`;
- `ProcessRunner` already de-duplicates event and start hooks;
- the host runner's repository persists module processes to the host
  `long_processes` table;
- continuations use the existing host process hook and rehydrate the stored
  module class; and
- commands returned by module process steps route through the module overlay.

The runtime merge validates that every module process class:

- extends `LongProcess`;
- is inside that module's namespace root; and
- does not conflict with a different host or module registration.

If the same class appears in the host base and a module catalog, registration
must either prove the entry metadata identical and de-duplicate it or fail. A
module may not replace host process metadata.

The same comparison applies against modules already wired for that host.
Metadata equality is strict catalog-data equality; 0.6.2 does not reinterpret
or reorder tag attributes. Validation of the complete current module catalog
happens before its listener constructors or process callbacks run.

A module container with no `LongProcessCatalog`, or with an empty catalog, is
valid for command/query/listener-only modules. That path does not resolve a
`ProcessRunner`. A non-empty catalog resolves the host's exact public runner
through `ConsumerRegistry::service_for()` and passes only non-duplicate entries
to the existing `register_process_entries()` function.

The module container may be dumped too. It uses the same 0.6.1 compiler pass to
materialize its own catalog; only the small catalog value crosses into runtime
module wiring.

## Dumped Container Compatibility

No step calls `register()`, `setDefinition()`, `addCompilerPass()`,
`findTaggedServiceIds()`, or another builder API on the host container.

Host runtime interaction is restricted to stable container operations:

- resolve public host services through `ConsumerRegistry::service_for()`;
- get the host `ProcessRunner`;
- optionally get public host middleware/services for the module bus; and
- read the host 0.6.1 catalog through the existing catalog-first host hook
  path.

This works for:

- a retained `ContainerBuilder`;
- a compiled `ContainerBuilder`; and
- a `PhpDumper` runtime `Container` with no definition/tag APIs.

## Dashboard and Discovery Attribution

`ConsumerRegistry::all()` remains the only top-level consumer source. It
contains the host and excludes its modules. Consequently:

- the consumer selector shows LMS once;
- no synthetic module prefix or ghost table appears;
- module command/event/process FQCNs appear in LMS audit, outbox, process, and
  trace records; and
- operational actions continue to resolve the LMS config and tables.

A future topology view may group module classes beneath their host, using
`modules_for()`, but that is not required for 0.6.2. Existing dashboard copy
that says one registered handle always equals one plugin or one code namespace
must be corrected to distinguish top-level consumers from modules.

Notifications remains outside this catalog. A raw WordPress subscriber may be
observed as a foreign subscriber, but it does not gain DDD ownership, config,
tables, or dashboard attribution.

## Failure Modes

### Missing or late host

`add_module()` throws when the host prefix is absent. `boot_module()` reports a
configuration error at `plugins_loaded:30`. It does not create a placeholder
consumer and does not wait until an arbitrary later action.

### Invalid or ambiguous namespace root

Registration throws when the root is empty, equals the host root, lies outside
the host root, or belongs more specifically to another registered consumer.
Whole-segment matching prevents `Tangible\LMSish` from matching
`Tangible\LMS`.

### Root conflict

Re-registering one root for the same host is idempotent before runtime wiring.
Registering that root for another host throws. Nested module roots are legal;
longest-root-wins remains deterministic.

### Module container wiring drift

A missing module bus/handler fails from the module container, not by falling
back to the host bus. A module `IDDDConfig` that is not the host object is a
wiring defect. The consumer-modules guide and conformance tests must verify the
binding.

### Host service visibility

If a dumped host makes a custom middleware or repository private, a module
cannot fetch it. Full host-native module execution is blocked until a normal
host release publishes that service under a stable public ID. Reconstructing
an equivalent stateful service is not a supported workaround, and runtime
container mutation is never the workaround. All four current fleet YAMLs use
public defaults, but each released dumped container still needs a contract
test for the exact IDs consumed by its sidecars.

### Duplicate hook boot

The WordPress facade guards module runtime wiring once per module root.
Registry replacement alone is not enough because constructing listeners twice
would add duplicate WordPress callbacks. Getter replacement is supported only
before WordPress `init` begins; a late facade call throws before changing the
route. The low-level `add_module()` pre-runtime constraint remains caller-owned
as documented under Registry API.

### Sidecar deactivation with live process rows

Stored rows contain the module process FQCN. Deactivating or removing the
sidecar can make rehydration fail. Before deactivation, operators must drain,
complete, fail, or migrate those rows. Renaming a process class likewise needs
a row migration or a compatibility alias.

### Base/module process conflict

A runtime module cannot override an entry from the host compiled catalog.
Identical duplication is de-duplicated; conflicting metadata fails during
module boot.

5. Prove a module command uses the module container while its handle exposes

intersect the 0.6.1 catalog-first code currently being implemented elsewhere.

## Minimal Post-0.6.1 Integration Patch

After rebasing this branch onto the completed 0.6.1 branch:

1. Keep the registry overlay and focused routing tests.
2. Add `ddd-wordpress/modules.php` with `boot_module()` and its once-per-root
   lifecycle guard.
3. Include that file from the winning framework loader.
4. Reuse the 0.6.1 shared `register_process_entries()` function. Do not copy its
   reflection, `#[Awaits]`, `#[StartsOn]`, or legacy tag handling.
5. At module `init:3`, read the module `LongProcessCatalog`, get the host
   `ProcessRunner`, validate module ownership, and pass entries to the shared
   function.
6. Eager-boot module event handlers/listeners through the existing
   `register_event_handlers()` helper.
7. Add one dumped-host regression test proving the host has no
   `findTaggedServiceIds()` or builder mutation API while module command,
   listener, and process registration all work.
8. Add lifecycle tests for missing host, duplicate module boot, catalog
   conflicts, and registration order.
9. Add a loader-order test in which a sidecar's 0.6.2 copy participates at
   `plugins_loaded:0`, the winner initializes at priority 1, the host boots
   after the winner and before the module (priority 10 in the current fleet),
   and `boot_module()` runs at priority 30.

Expected intersection with 0.6.1 is limited to the WordPress hook/loader files
and the new catalog API. No compiler-pass file should need a semantic change.

## Verification

Focused spike tests must prove:

- longest-root routing sends a class under
  `Tangible\LMS\Extension\SuperTrace` through the module bus;
- the module routing handle uses the exact LMS config object;
- `service_for()` returns the host's exact middleware/repository objects and a
  module bus built from those objects still resolves its terminal handler in
  the module container;
- host namespace classes still use the host container;
- module event actions use the LMS prefix;
- top-level discovery contains LMS but not a second module consumer;
- `modules_for('tgbl_lms')` enumerates the overlay for the future catalog
  layer;
- a missing host fails;
- an external namespace such as `Tangible\Notifications` cannot be attached;
  and
- the opaque host container receives only the expected `get()` calls and has
  no builder/mutation dependency.

The spike includes a real `PhpDumper` test for the get-only cross-container
bridge and shared object identity. After the 0.6.1 rebase, extend that proof to
the full WordPress facade and runtime process overlay.

## Open Questions Deferred From the Spike

1. Should the final API return a distinct `ConsumerModuleHandle` while keeping
   `owner_of()` source-compatible through an interface, or retain an ordinary
   `ConsumerHandle` plus overlay metadata? The spike deliberately chooses the
   smaller, source-compatible form.
2. Should the known fleet service IDs become a documented framework contract,
   or should each sidecar declare a versioned host-service manifest? The
   transaction ID already differs: LMS and Quiz use their host Doctrine
   middleware, while Cred and Datastream use the framework middleware.
3. Resolved for 0.6.2: `boot_module()` keeps a host-scoped runtime metadata
   ledger, validates the current catalog against the host and previously wired
   modules, and passes only new entries to 0.6.1's
   `register_process_entries()`.
4. Should the dashboard eventually show module grouping metadata, or is the
   module FQCN within the host trace sufficient? This is a dashboard product
   decision, not a persistence requirement.
