# 0.3 Spec — TraceContext, Correlation, and the Causation Metamodel

**Status:** SPEC — approved direction, not yet built. Target: 0.3 (flag-day
release; touches every CorrelationContext reference and the envelope contract).
**Provenance:** multi-day design session 2026-07-16 → 2026-07-18, on top of the
0.2.4 execution-frames/ignition work (PR #14). Forerunner pieces ship in 0.2.5
(see Migration).
**Audience:** framework maintainers and future AI sessions. Consumer developers
never touch these names — the machinery is ambient by design.

---

## 1. Diagnosis — why this redesign exists

The journey of a message — *which story is this, who caused me, where in the
story am I* — currently exists as **five projections of a value that was never
written down**:

| Home | Form |
|---|---|
| `CorrelationContext` | four loose static fields (correlation, causation pair, sequence) |
| the wire | three `__`-keys smeared into the payload bag |
| four tables | columns named differently on each (`causation_id`+`causation_type`, `command_id`, `ignited_by_event_id`) |
| event objects | two private slots, mutated post-construction by `stamp_journey()` |
| `TransportEnvelope` | three readonly properties |

Every ritual in the subsystem — `restore_context()`, `stamp_journey()`, the
`__`-key smear, arm/consume causation — is hand-copying fields of this nameless
thing between its homes. Three concrete diseases:

1. **No first-class value.** You cannot name the movers well when the thing
   they move has no name (this is why every naming attempt on
   `restore_context` felt wrong).
2. **Two generations of idiom in one class.** The causation slot is a
   *one-shot token* (arm → first consumer takes it → everyone else finds it
   empty). 0.2.4 taught the codebase *scoping* (`with()`, the sealed
   `with_process()` bracket, typed frames). Every known causation hole is a
   one-shot-token failure a scope wouldn't have.
3. **Facts learn identity by post-birth mutation.** Events are immutable
   records by doctrine, yet `stamp_journey()` writes identity into private
   slots after construction, at four call sites that must each remember to.

## 2. The fix, in one sentence

> While work executes, brackets open **`Correlation`** scopes carrying
> **`TraceContext`** values; at every boundary a **`IntegrationEnvelope`** moves the
> context; every table row is a *projection* of it; and later, **`Tracer`**
> reads the projections back into a **`Trace`**.

Every word does one job. The rest of this document is that sentence, expanded.

## 3. Ubiquitous language

This subsystem has its own domain model — the **accountability of work** — and
these are its terms. The ontology (Acts/Facts/Trajectories) is a **meta-model**:
consumer commands are *instances of* Act, the way businesses live inside
accounting without using its vocabulary.

| Term | Meaning |
|---|---|
| **Act** | A command pass — the atomic, audited unit of work. Recorded in `command_audit`. |
| **Fact** | A *published* integration event — recorded, immutable, part of a story. An `IntegrationEvent` is the **genus** (an event *capable* of crossing boundaries); *fact* is the **status it acquires at publication**. Envelopes only ever carry facts. Domain events that never publish are events that never became facts. |
| **Trajectory** | A LongProcess — a named lifecycle that frog-leaps between Acts across time. Recorded in `long_processes`. BehaviourWorkflows are trajectory *overlays*, not a kind (ruling). |
| **Story** | Everything sharing one `correlation_id`. Prose/ops word; support says "case". |
| **Cause** | `(id, kind)` — the node currently executing, which everything born inside it will record as its parent. |
| **Agent** | *Who started a story.* Root annotation (`source`/`source_id`), never a node. Meaningful **only at roots** — on interior rows the source column records worker SAPI noise. |
| **TraceContext** | The immutable propagated value: `{correlation_id, ?Cause, sequence}`. W3C-exact name — the Trace Context spec is a *propagation* standard (write-side native). |
| **Correlation** | The runtime facade — static access, scope brackets, the sequence counter. |
| **Trace** | The assembled read-side artifact (tree + warnings). |
| **Tracer** | The read service that assembles a Trace from the projections. |

**The tense rule** (decides which register a name belongs to): you
**correlate** as you go — present tense, the live activity of tying things
together. You **trace** after the fact — following the tie back. Write side
speaks correlation; read side speaks trace; the propagated value takes the
standard's name because propagation is what the standard defines.

### Naming rulings (so they are not re-litigated)

| Decision | Ruling | Why |
|---|---|---|
| Facade name | `Correlation`, NOT `CorrelationContext` | Old god-class name never reused — old and new must be distinguishable in a diff |
| Value name | `TraceContext` exactly | Letter-for-letter the W3C/OTel term = free documentation. Near-misses (`TracedContext`) are worse than distant names: look like typos, grep against nothing |
| `Journey` | RETIRED entirely | Marketing-deck whiff; no method, field, or doc keeps it |
| `Span` | rejected for the cause pair | OTel spans are recorded objects with duration/attributes; ours is a reference. Over-promise |
| `TraceSlice`/`Bite`/etc. | rejected | The value contains no trace *material* — it's coordinates, not a sample. Whimsy belongs in UI copy (see: 👻), not the type system |
| `Story`/`StoryContext` | prose only | The value is coordinates *within* a story, not the story; and coining a private synonym for a standard term taxes every reader |
| `CorrelationTimeline` | renamed `Tracer` before ever built | Read-side service must not wear a write-side word; "Timeline" contradicts the topology ruling (order comes from causation, not time) |
| envelope name | **`IntegrationEnvelope`** (owner ruling 2026-07-19, revising an earlier `FactEnvelope` pick) | The envelope is MACHINERY, and machinery names stay in their family: IntegrationEvent/Behaviour/Listener/action/**Envelope** — one grep yields the whole transport story. `FactEnvelope` smuggled an ontology word into a machinery name; the "only facts ride" lesson lives in the docblock. Fact vocabulary keeps its homes: `Kind`, trace surfaces, prose |
| `ConsumerRegistry` | keeps its name (not `DDDRegistry`) | The `DDD` marker earns its keep only on names imported into foreign namespaces (`DDDConfig` yes — it sits among consumer imports); framework-internal names should say what they hold |
| field `correlation_id` on TraceContext | stays (not W3C `trace_id`) | Columns are forever; we borrow the class name and semantics, fields align with schema |

## 4. The values

```php
final class TraceContext {
    public function __construct(
        public readonly string $correlation_id,
        public readonly ?Cause $cause = null,       // the node we're inside; null = flat context
        public readonly int    $sequence = 0,
    ) {}

    public static function root(): self {
        return new self(Uuid::v4());                 // a new story; no cause, position 0
    }

    // ── derivations: SAME story, new "you are here" marker ──────────
    public function for_fact(string $event_id): self {
        return new self($this->correlation_id, new Cause($event_id, Kind::Fact), $this->sequence);
    }
    public function for_act(string $command_id, ?string $label = null): self {
        return new self($this->correlation_id, new Cause($command_id, Kind::Act, $label), $this->sequence);
    }
    public function for_trajectory(string $process_id, ?string $label = null): self {
        return new self($this->correlation_id, new Cause($process_id, Kind::Trajectory, $label), $this->sequence);
    }
}

final class Cause {
    public function __construct(
        public readonly string  $id,      // ALWAYS a borrowed id: command_id | event_id | process id. Never minted.
        public readonly Kind    $kind,
        public readonly ?string $label = null,   // class name; diagnostics ONLY (exception messages).
                                                 // Never persisted, never serialized — stripped at every boundary.
    ) {}
}

enum Kind: string { case Act = 'act'; case Fact = 'fact'; case Trajectory = 'trajectory'; }
```

Notes:

- **`Kind` lives write-side**, next to `Cause` (it types the `causation_type`
  column). The read-side `TraceNode` imports it — the ontology gets exactly one
  canonical home in the type system.
- **At rest, `Kind` maps to legacy column values** (`'integration_event'`,
  `'long_process'`) in the projection layer. No schema change.
- **`label` is the frames' ghost**: the 0.2.4 execution frames stored class
  names so guard exceptions could *name both parties*. Cause stores ids; the
  optional label preserves the error-message quality. It exists exactly where
  the frames existed — in-process, for one scope — and nowhere else.
- The derivations change ONLY the cause slot. Same story, same position.

## 5. The facade

```php
final class Correlation {
    private static ?TraceContext $current = null;
    /** @var array<array{ctx: ?TraceContext, seq: int}> */
    private static array $stack = [];
    private static int $sequence = 0;

    public static function current(): TraceContext {
        return self::$current ??= TraceContext::root();     // flat context lazily mints a new story
    }

    public static function within(TraceContext $ctx, callable $work): mixed {
        self::$stack[]  = ['ctx' => self::$current, 'seq' => self::$sequence];
        self::$current  = $ctx;
        self::$sequence = $ctx->sequence;      // adopt the incoming story's position
        try     { return $work(); }
        finally {
            $frame          = array_pop(self::$stack);
            self::$current  = $frame['ctx'];   // restore the WHOLE enclosing context…
            self::$sequence = $frame['seq'];   // …including its counter (see §5.1)
        }
    }

    public static function next_sequence(): int { return ++self::$sequence; }
}
```

That is the entire runtime: one immutable pointer, one stack, one counter.

### 5.1 Why the stack snapshots the counter (the cross-story wake)

In the common case every scope on the stack shares one correlation and the
stack is effectively scoping the *cause*. But there is one place correlation
legitimately changes mid-stack: **the wake bracket, when the awaited fact comes
from a foreign story.**

Example: `DeadlineBoundCompletionProcess` is born from an admin action (story
A) and suspends awaiting `EarningIssued`. Days later the learner completes a
course on their own initiative (story B). B's fact wakes A's saga:

```
drain of EarningIssued          current: { corr: B, cause: FACT evt_x }
  └ wake bracket for saga #191  current: { corr: A, cause: TRAJ 191 }   ← STORY SWITCH
      └ saga dispatches act     current: { corr: A, cause: ACT cmd_y }
```

The saga's post-wake acts belong to *its own* story; when its bracket exits,
B's drain ambient must come back intact (the fact may wake more sagas). This is
why:

- the stack stores **full contexts**, not ids;
- the **sequence counter is per-scope snapshot**, not a global max — while A
  runs, facts take A's next position; on exit, B's counter is back where it
  was. (A global counter lets two stories corrupt each other's positions.)
- read-side consequence: "woken by" cross-links can point *outside* the trace
  being rendered — a **portal** between stories, derivable (process row and
  fact each know their correlation), rendered as a link, never an edge.

### 5.2 Legal stack grammar

Depth is capped at 3 by the guards. The only legal stack shapes are the paths
through the causation matrix — **fact → trajectory → act, each layer optional,
order fixed**:

```
[ ]                  flat            [FACT, ACT]         listener's command
[FACT]               draining        [FACT, TRAJ]        ignition
[TRAJ]               cold saga wake  [TRAJ, ACT]         saga frog-leap
[ACT]                root command    [FACT, TRAJ, ACT]   ignition's first step   ← maximum
```

The scope stack at any instant spells a grammatically valid sentence of the
ontology; the guards are its grammar checker.

## 6. The three brackets

Every green edge in the causation matrix reduces to one rule: **a node's
parent is `Correlation::current()->cause` at its birth.** Four stamp sites, one
mechanism, zero arming rituals.

### 6.1 The drain bracket (listener ceremony / ignition entry)

```php
add_action($config->integration_action('earning_issued'), function (array $wrapped) use ($listener) {
    $envelope = IntegrationEnvelope::unwrap($wrapped);
    $event    = EarningIssued::from_payload($envelope->payload);

    Correlation::within(
        $envelope->trace_context()->for_fact($envelope->event_id),
        fn () => $listener->handle($event)
    );
});
```

Replaces `restore_context()` (init + set_sequence + arm-one-shot). The fact is
the ambient cause *for the whole closure* — a listener dispatching three
commands gives all three `causation = this event`. The fat-listener hole stops
being expressible.

### 6.2 The act bracket (CorrelationMiddleware)

```php
public function execute($command, callable $next) {
    $enclosing = Correlation::current();

    if ($enclosing->cause?->kind === Kind::Act) {
        throw new CommandDispatchedInsideCommand(get_class($command), $enclosing->cause->label ?? $enclosing->cause->id);
    }

    $command_id = $this->ids->generate();
    $this->audit->record($command, $command_id, $enclosing);   // row causation = ENCLOSING cause

    return Correlation::within(
        $enclosing->for_act($command_id, get_class($command)),
        fn () => $next($command)
    );
}
```

The guard becomes a *read* (`cause->kind`); `mark/clear_command_frame` die.
Facts published inside the handler take their raiser edge from
`current()->cause->id` — `set_command_id()` dies.

### 6.3 The wake bracket (ProcessRunner::with_process)

```php
private function with_process(LongProcess $process, callable $work): mixed {
    return $this->with_process_lock($process, function () use ($process, $work) {
        $ctx = new TraceContext($process->correlation_id());       // the row is the TEMPORAL carrier
        return Correlation::within(
            $ctx->for_trajectory((string) $process->get_id(), get_class($process)),
            $work
        );
    });
}
```

The old `CorrelationContext::with($id, $fn)` dies here — it took a *string*
(correlation only), which is exactly why the runner needed the per-command
`set_causation`/`clear_causation` arming loop in `dispatch_commands()`. The
full value carries the trajectory-cause; the loop goes away.

### 6.4 Ignition absorb (ProcessRunner::start)

```php
$cause = Correlation::current()->cause;

if ($cause?->kind === Kind::Act)        throw new ProcessStartedInsideCommand(...);
if ($cause?->kind === Kind::Trajectory) throw new ProcessStartedInsideProcess(...);   // NEW guard

$process->initialize_lifecycle(Correlation::current()->correlation_id, $steps);

if ($cause?->kind === Kind::Fact) {     // manual ->start() inside a drain: record the truth
    $process->mark_ignited_by($cause->id);
    $process->mark_source('event');
}
```

## 7. The causation matrix (target state: no cell lies)

| Causator → Causee | Status | Storage / behavior |
|---|---|---|
| Fact → Act | recorded | `command_audit.causation_id` + type (scoped, covers commands 2..n) |
| Trajectory → Act | recorded | same columns, cause from wake bracket |
| Act → Fact | recorded | `outbox.command_id` from `current()->cause->id` |
| Fact → Trajectory | recorded | `long_processes.ignited_by_event_id` (`#[StartsOn]` *or* absorb) |
| Act → Act | forbidden | throws `CommandDispatchedInsideCommand` (0.2.4) |
| Act → Trajectory | forbidden | throws `ProcessStartedInsideCommand` (0.2.4) |
| Trajectory → Trajectory | **forbidden (NEW)** | throws `ProcessStartedInsideProcess`. Process-in-process is spelled parent-step → **Act** → **Fact** → `#[StartsOn]` child: every hop recorded, and the fan-out decision is an *audited* command. A direct spawn edge would skip an audited moment ("coordinators sequence, commands act"). The shelved `spawn`/`IWork` design stays shelved — the thing it wanted is already expressible, better |
| Trajectory → Fact | **forbidden (NEW)** | publish-path guard: `cause->kind === Trajectory` and no act frame → throw. Steps sequence commands; they never announce |

Origins (roots) are annotations, not edges: `source`/`source_id` on audit
(agent stamp — the only "who" record; render at roots only), `source` on
processes, null `command_id` on human-announced facts (`wp ddd announce`).

## 8. Dissolution table — where every old piece goes

| Old | Fate |
|---|---|
| `CorrelationContext::$correlation_id/get/set/init/peek` | → `TraceContext->correlation_id` via `Correlation::current()` |
| one-shot `$causation_id/$causation_type` + `set/clear_causation` | → the ambient cause. DELETED — no arming, no consuming |
| `$command_id` + `set_command_id` | → `current()->cause->id` when kind=Act. DELETED |
| `$command_frame`/`$process_frame` + mark/clear (0.2.4) | → `cause->kind` reads + `label` for messages. The frames were the cause slot in a trenchcoat; shipping them in 0.2.4 was scaffolding that live-proved the guard semantics |
| `$scope[]` stack of correlation ids | **the one survivor, promoted**: stack of full `{ctx, seq}` snapshots in `within()`. Teardown special-case (`if empty → reset`) and per-leave `command_id` clearing both fall away structurally |
| `depth()` | dead (deadcode sweep); does not return |
| `with($correlation_id, $fn)` | → `within(TraceContext, $fn)`; only production caller was `with_process`. The `EventPipelineTest` ambient-wipe quirk dies with it (full-value restore) |
| `reset()` | → `Correlation::reset()` (test seam) |
| `TransportEnvelope` | → `IntegrationEnvelope` (0.2.5 rename): `{payload, event_id, TraceContext}` + symmetric `wrap()`/`unwrap()` (smear moves in from OutboxProcessor). Cause is NOT serialized — raiser edge is already at rest in `outbox.command_id`; the drain derives a fresh cause |
| `restore_context()` / (`restore_journey`) | DELETED — replaced by composition: `unwrap()` → `within($env->trace_context()->for_fact(...), $drain)`. Restoring was never one operation; it was parse-then-scope |
| `stamp_journey()` + journey slots on `IntegrationBehaviour` | DELETED — facts become genuinely immutable; identity lives on the envelope. The trait shrinks to codec + `integration_action()` |
| `CorrelationMiddleware` | survives — and the name upgrades from coincidental to exact (it operates the `Correlation` facade) |
| `CommandAuditMiddleware` | likely merges into the act bracket (see Open Questions) |

## 9. Sequence — the honest note

`sequence` is a per-story Lamport clock: happens-after along causal chains, NOT
a total order (sibling drains resuming from the same position mint duplicates —
by design; topology comes from causation, timestamps are display). Two rulings:

- It is the **weakest write-side citizen**: guards don't need it, the audit
  obligation arguably doesn't — its justification lives almost entirely on the
  read side. Kept because retrofitting ordering is impossible by definition,
  but if `TraceContext` ever sheds a field, it's this one.
- It is what *forces* the stack to snapshot everything (§5.1).

## 10. Statics ruling

Mutable static state is the liability; static *access* to immutable values is
fine and universal (OTel context, .NET `Activity.Current`, logging MDCs).
`Correlation` is the pattern's static half and the **single worker-mode seam**:
under FrankenPHP/Swoole the holder's storage becomes context-local without one
call site changing. Threading `TraceContext` explicitly through signatures is
the tramp-data anti-pattern; don't. `ConsumerRegistry` stays a plain static
forever (process-lifetime, like an autoloader).

## 11. Read side (dddash / CLI / tests — optional consumers)

The write side is designed **as if no reader exists** — every concept above
justifies itself on correctness or audit grounds alone. Reading is derived:

- `Tracer::trace($correlation_id): Trace` — gather (per `consumers()`, per
  family: audit/outbox/processes, `table_reachable`-guarded, PHP-side merge —
  survives consumers moving to separate DBs) then stitch (pure PHP:
  `TraceStitcher`, `TraceNode`, `Trace{roots, index, warnings}`).
- Roots are discovered **structurally** (parentless node), never by table —
  three spawner kinds (root act / cold trajectory / human-announced fact) cost
  zero extra queries.
- Wakes are cross-references, not edges (a fact's parent is its raiser; the
  saga it wakes is a reader). Cross-story wakes render as portals.
- No central correlation table, ever: identity is coordination-free (UUID),
  context is process-local or carried, participation is derived. An origins
  table may someday exist as a *derived, rebuildable* listing index — never a
  write-side check-in.

## 12. Meta-model discipline

The kind set is **closed at three**. Admissions are rulings, not drift. Prior
rulings that kept it closed: workflows are trajectory overlays; agents are root
annotations; process-in-process spells through existing kinds. The pressure to
add a fourth kind will recur — the answer is this section.

(Historical note: the `$scope` docblock predicted this design — "frames can
later carry a span id + kind (command|process|workflow)". The ontology
corrected the guessed taxonomy: *workflow* demoted to overlay, *fact* promoted
— being inside a drained fact is as much a position as being inside a command.)

## 13. Migration

**0.2.5 forerunners (additive, each independently shippable, same semantics
the 0.3 cells would give):**

1. Scoped causation (re-arm per dispatch or scope-based) — closes the
   fat-listener hole
2. `ProcessRunner::start()` absorb — closes manual-start-in-drain
3. `ProcessStartedInsideProcess` guard (+ publish-path Trajectory→Fact guard)
4. `TransportEnvelope` → `IntegrationEnvelope` rename + `wrap()` move (4 files: 3
   framework + datastream's `DeliveryOutboxPublisher`)
5. `Tracer` + stitcher (read-only, retroactive, no schema change)

**0.3 proper (flag-day, all consumers lockstep):** `TraceContext`/`Cause`/
`Kind` + `Correlation` facade + brackets; delete the dissolved pieces (§8);
envelope-first identity (delete `stamp_journey`). Rides the same break as
`DDDConfig`/stamp-collapse (see 0.2.5 consumer-update plan).

**Do not build 0.3 ahead of a consumer need.** This spec exists so the
intermediate moves stay cheap and aligned — the puzzle can know its final
picture long before assembly. (Governor: every abstraction here corresponds to
a shorter debugging session or an impossible production failure; the spec
itself is the only artifact whose absence was costing us — re-derived three
times from transcripts before being written down.)

## 14. Appendix — the rest of the 0.3 pile (shelved designs, so they are not lost)

Designs discussed and shelved during the 0.2.4 arc, distinct from TraceContext
but riding the same release horizon. Each was scoped in conversation; this is
the durable index.

1. **`SelfHandlingCommand`** (name ruled; over `SelfContainedCommand`; the
   `IHandlesItself` working title retired). Kills the command/handler two-class
   ceremony while keeping the middleware onion: the command declares a
   `protected function handle(...$deps): void` whose parameters are hydrated by
   reflection from the container (method injection, like Symfony's
   ArgumentResolver — note Symfony has no `container->call()`; Laravel does).
   `protected` + PHP 8.1's `setAccessible` no-op means only the bus middleware
   (`SelfExecutingCommandMiddleware`) can invoke it — no manual calls. One
   consumer-stamped base (or registry-resolved under the 0.2.5 scheme). The
   handler-as-separate-class remains legal for multi-dependency-heavy cases.
2. **The receipt rule.** `handle()` MAY return a *receipt* (scalar/DTO verdict
   for the transport — e.g. the datastream retry verdict), MUST NOT return
   domain objects, and nothing downstream may *depend* on the return. Resolves
   the long-standing "void handle() shtick" discomfort without opening the
   query-through-command door.
3. **"arc" — closure-map process steps.** Alternative authoring for
   loop-shaped processes: steps as an ordered closure map instead of
   source-order reflected methods. Addresses the WorkflowHandler
   phase-`switch` kookiness while keeping the code-in-one-place virtue. Two
   execution geometries acknowledged: stage-shaped (LongProcess methods) vs
   loop-shaped (WorkflowHandler phase machine).
4. **Un-existing the `ISagaBehaviour` trio** (retry / stop /
   user_triggered_retry) into real sagas, keeping Behaviour config objects as
   data. Known gaps to build first: `Await::clock` (wall-clock await mechanism)
   and a `UserProfileUpdated`-class fact for user-triggered resume.
5. **`IWork` / `Result::spawn` (child sagas) — KILLED** by this spec's §7
   ruling: process-in-process is spelled parent-step → Act → Fact →
   `#[StartsOn]` child. The spawn edge would skip an audited moment.
6. **Multi-item behaviour topology** (VerdictMap / IOutcomeInterpreter —
   plurality-aware behaviours). Half-realized by the WorkflowHandler ledger
   runner; full design capture in project memory
   (`multi_item_behaviour_topology`) and the framework topology doc on
   datastream's `v3-ddd`.
7. **WorkflowRunner / behaviours-as-stored-intents** (2026-07-18). The
   BehaviourWorkflow tick-command (`ProcessEndpointResponseBehaviourCommand`
   hammering the workflow forward) is the causation grammar inverted: an Act
   driving a coordinator — a clock wearing a command costume. Target shape: an
   opaque WorkflowRunner (peer of ProcessRunner — load row → ask aggregate for
   next intent given last verdict → dispatch through the NORMAL bus →
   interpret → persist). Behaviours don't get a privileged runtime with
   repo/service access — "command-like access" is just a command; behaviours
   BECOME intents. Route A: per-behaviour intent commands via thin translation
   (the IntegrationListener pattern applied to config). Route B: behaviour
   configs ARE serialized commands (ctor-as-schema + total codec — announce
   already proved JSON-constructibility); admin-authored sequences = stored
   intents + verdict logic. Constitutional note: a dispatching workflow is
   ontologically a Trajectory — kind set STAYS closed at three, no
   `behaviour_workflow` causation_type. Likely endgame: BehaviourWorkflow
   collapses into a data-driven LongProcess ("arc" closure-map as the bridge,
   ISagaBehaviour un-existing as the migration, multi-item topology as the
   ledger). Completed 2026-07-18 by the "behaviours are executed by domain
   services" observation: that's the normal anatomy of every command (handler
   thin, delegates to services) — the per-behaviour unit is a
   SelfHandlingCommand whose ctor IS the stored config (JSON-hydratable),
   whose handle(...$services) is the domain-service call site, and whose
   return is the verdict — the receipt rule verbatim. CONVERGENCE: items
   1 (SelfHandlingCommand), 2 (receipt rule), 3 (arc), 4 (ISagaBehaviour
   un-existing), 6 (multi-item topology), and this one are ONE design —
   data-driven trajectories dispatching self-handling stored intents that
   return verdict receipts. The shelf was a disassembled machine.
   **STATUS (owner ruling 2026-07-18): SHELVED, not scheduled.** The
   convergence is an observed design fact; whether to BUILD the machine
   (WorkflowRunner, ISagaBehaviour-as-process, the rest) is undecided —
   a possible future change, not a 0.3.x commitment. Nothing in the 0.3
   metamodel work depends on it.
8. **Stamp collapse / `DDDConfig`** — scheduled 0.2.5, not 0.3 (additive):
   framework-shipped `DDDConfig` concrete + registry-resolved bases
   (`owner_of(static::class)` for `container()`/`prefix()`), per-handle
   JsonLifecycleValue renderer, parameterized inflector, scaffolder stops
   stamping 8 classes (→ boot declaration + yaml). Consumers migrate via
   hollow-shim bases, independently, no lockstep. Foundation
   (`ConsumerRegistry::owner_of()`, namespace roots, label-clobber fix) already
   shipped on `fix/0.2.4` (`bd1cc60`).
9. **The act receipt / write footprint** (2026-07-19, post-0.4.0 — the
   return-value conversation widened). Extends the receipt rule (item 2):
   `send()` could return a `Receipt { verdict, footprint }` where the
   *footprint* is the act's write manifest — the missing write-side sibling
   of `EventsUnitOfWork`. Shape: keyed by canonical aggregate name, values
   are LISTS of `(id, op)`, ops = `created | updated | deleted` (three
   verbs, or the manifest lies by omission); a `primary()` entry marks the
   command's subject. Ids and names ONLY — never attributes (an attribute
   makes it a read model and the discipline collapses). Harvested at the
   repository chokepoint (`save()`/`delete()` — where `insert_id` already
   lands), collected per-act in a value scoped by the act bracket — the
   post-0.3 `within()` lifecycle makes this safe where it would previously
   have been another rotting ambient static; the no-nesting guard gives
   one act = one footprint. Only `Aggregate` subtypes report: the footprint
   tracks domain state, not storage (denormalizations/counters/logs are
   invisible). Rules, restated for the richer object: (a) ids+names only;
   (b) edges may read it, domain may not — a handler chaining on another
   command's footprint is the same smuggled workflow as chaining on a
   returned id; continuation is spelled fact → listener; (c) descriptive,
   not contractual — `primary()` is the only stable promise; asserting the
   full write-set couples edges to handler internals.
   **What it unlocks** (each a separate decision): transport redirects via
   `primary()` uniformly (the MySQL-auto-increment answer to the purists'
   pre-minted ids); footprint persisted on `command_audit` COMPLETES the
   act record (story, parent, params, facts, state touched) — squarely the
   observability track; the *aggregate biography* (entity-keyed footprint
   index → dddash's second axis: story view AND entity view; "who touched

   a toy); compensation checkpoints become projections of step receipts
   (undo targets harvested, not hand-recorded); idempotent replay
   (receipt stored on the audit row; a redelivered `command_id` answers
   from the stored receipt, Stripe-style); a formalized verdict enum

   investigation); the doctor's bookkeeping-balance check (writes with no
